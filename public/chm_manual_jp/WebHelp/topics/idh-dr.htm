エラー報告

バグレポートを参照するために、1つの特定のプロジェクトの名前を選択する必要がありますので誤差が各プロジェクトに行われる報告（チーム、部門での連結レポー場合がない）

 

 

- Department、Division、Team、Projectは検索プロジェクトを支援する。エラーレポートを表示するには一つ特定のプロジェクトを選択する必要があります。

- Units of time: 単位時間は、レポートを表示したいです、日、週、月の値を持ちます。

- Bug: エラーの単位、2値があります：Number of bug- エラーの数に従って計算、Weight bug- 加重エラー に従って計算。

- Report type: レポート類、３値があります： All - すべてのチャートを表示、By summary- サマリーをかけてエラー統計チャートこと、By time - 時間をかけてエラー統計チャートこと。

- 間隔ビューレポート: リストから選択することができます、または開始日と終了日を入力する。

 

エラー報告の種類:

①エラーが原因で発生したエラーの根本的な原因でレポート通常、思いやりの理由が生じました:

次のようにすべての原因は6グループに収集されています:

値

説明

Requirements

不完全であるかは不明に起因する要求エラー

Design Error

以下からの要求とデザインの違い,または設計によって不完全、不明です

Code Error

設計ドキュメントまたはリクエスト作られた、または間違ったコードの構文で、内部構造のコードの違い

Test Error

不適切な設計のテスト（要件やデザインの違か、または間違った手順を実行して、設定ミス、不正なデータ入力エラーからログを結果として不正

Configuration

環境や不正なコンフィギュレーションデータに起因する誤差

Existing bug

エラーが過去に発生し、現在は再び表示されます

 

②状況によってバグを報告

エラーステータスは、次の状態が含まれています。

ステータス

説明

New

チケット新しく作成されたとコミットしていない人

In progress

チケットは実施されています

Resolved

チケットは処理され、テストやレビューを待っているされています

Feedback

ユーザーが確認することができますか不明瞭なテストが確認以上修正する必要があります

Closed

チケット編集または完全に行われる

Reject

このチケットは受け付けていません。または不正確

 

③エラーを見つける人による報告する ( Created by- バックログ上とRedmineのデフォルトは、Creatorは、エラーログを探していました。顧客のエラーによる検索は、BSEは、エラー・ログが顧客を助けることになります)

 

④エラーを修正する人による報告する( Assignee)

⑤エラーの原因となったか、エラーが誰に属している人による報告する( Who made defect)

⑥時間プロジェクトによって発見されたエラーと顧客への正式なハンドオーバ後に発見されたとエラーによる報告する ( Bug và Bug after release)

報告書は最後まで最初から累積誤差を示しています。

グリーンラインはインクリメンタルプロジェクトチームは、障害を発見しました

ブルーラインは累積誤差は、顧客への正式なハンドオーバ後に発見されました

黄線ラインは累計エラーシャット

ブルーラインは日付で見つかったエラーの数
